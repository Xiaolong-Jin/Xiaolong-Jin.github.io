又见 Swift

本文主要从公司团队的角度出发，讲解一些 Swift 中比较难以理解的概念。
因已经零星接触过一些 Swift 的知识，所以不会全面罗列。

访问级别控制

Swift 中共有五种访问权限控制，分别是 open,public,internal,fileprivate和private，其访问权限依次递减。

open: 模块（Target）内外都可以访问，只能修饰 class 和 class 内部的 property/func。其修饰的 class 在模块外部可以被继承，其修饰的 property/func 可以被 override。
public: 模块内外都可以访问，但其修饰的 class 仅在模块内部可以被继承，其修饰的 property 和 func 仅在模块内部可以被 override。（如果在模块内部也不想被继承/override），可以添加 final 修饰。
internal: 默认类型，模块内部可以访问。
fileprivate: 源文件内可以访问。
private: 修饰类型(protocol/class/struct/enum)则该类型只能在源文件内访问（同 fileprivate）；修饰成员(变量/方法/嵌套类型)则该成员只能在所属的类型（包括extension）内访问。

几条约定规则：

  1. 类型的成员的访问级别不会高于类型的访问级别（包括嵌套类型）。例如：一个 internal 级别的类内成员不会是 public/open 级别的。
  2. 类型的成员的默认访问级别和类型的访问级别相同，且最高为 interval；private 修饰的类型，其成员的默认访问级别为 fileprivate。
  3. 实例的的访问级别不能高于其声明的类型的访问级别。例如：一个 private 级别的类的实例对象只能是 private 级别的。
  4. 属性的 setter 的访问级别可以低于 getter 的访问级别。例如：一个 internal 级别的属性，可以添加 private(set) 修饰，使其 setter 方法只在类型内部使用，对外是 readonly 的。
  5. 协议中必须实现的成员的访问级别和协议本身相同，子协议的访问级别不高于父协议。
  6. 一个类的访问级别不能高于（默认等于）其父类和实现的协议中最低的一个。
  7. extension 及其中的成员的访问级别符合规则1，且 protocol 的 extension 的访问级别不可以显示声明。
  8. Tuple 的访问级别是其内各个元素的最低访问级别。（自动推导，无法修改）
  9. func 的访问级别是返回值和参数中的最低访问级别，且与默认访问级别 internal 不符的时候需要显示声明。
  10. Enum 中 case 成员的访问级别等于 Enum 本身，且不能单独设置。Enum 的关联值、原始值的访问级别不能低于 Enum 本身。

注意：约定1 中的“不会”，而非“不能”。类型的成员可以显示声明更高的访问权限，但是由于类型的访问权限限定，类型的成员的更高的访问权限并不能生效。

Property

Swift 中的成员属性分为 计算属性 和 存储属性 两类。

存储属性 在运行时会被分配相应的内存空间来进行存储；而 计算属性 没有实际的储存空间，类似 OC 中没有生成 iVar 的 property。

Swift 中可以通过 willSet 和 didSet 方法来监听 property 的改变。
注意：willSet/didSet 不能和 set/get 一起使用，其内部的逻辑可以直接放到 set 方法中。

关于 oldValue 和 newValue:
willSet 和 set 中可以通过 get 获取旧值，newValue 获取新值。
didSet 中可以通过 get 获取新值，oldValue 获取旧值。

上述例子中，先调用了一次 get, 因为 didSet 中用到的 oldValue，这个值需要在整个 set 动作之前先存储起来。如果不实现 didSet 方法，就不会调用 get 了。

必须写成

Swift 是一门安全的语言，String 和 String? 是两种不同的类型，String? 的本质是 Optional<String>。通过 Optional 的定义，我们可以发现，它是一个实现了 ExpressibleByNilLiteral 协议的泛型枚举。

Swift 中的字面量类型有

  • ExpressibleByNilLiteral
  • ExpressibleByFloatLiteral
  • ExpressibleByArrayLiteral
  • ExpressibleByStringLiteral
  • ExpressibleByBooleanLiteral
  • ExpressibleByIntegerLiteral
  • ExpressibleByDictionaryLiteral
  • ExpressibleByUnicodeScalarLiteral
  • 以及稍稍特殊的 ExpressibleByStringInterpolation

通过实现 ExpressibleByNilLiteral 协议，我们可以让一个 String 类型的变量直接被 nil 赋值。
